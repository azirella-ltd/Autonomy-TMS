# Complex_SC Agent Call Review

## Supply chain topology for Complex_SC
- `create_regional_sc_config.py` seeds the Complex_SC network with 10 finished goods, three market demand nodes, three distribution centers, two plants, and 30 suppliers across the three regions, wiring suppliers→plants→distribution centers→market demand through explicit lane records with deterministic information and material lead times.【F:backend/scripts/create_regional_sc_config.py†L82-L240】
- These lanes are consumed by `_build_lane_views`, which normalises node keys, fills in missing market-supply sources, and runs a topological sort over the shipment graph to derive a downstream→upstream node ordering alongside shipment and order adjacency maps.【F:backend/app/services/mixed_game_service.py†L366-L490】

## Game start-up
- When `MixedGameService.start_game` is called, it regenerates `node_policies`, validates the lane list from the configuration, reuses `_build_lane_views`, and records the sorted node order (`node_sequence`) plus derived node types back into the game configuration. Each node’s initial inventory, base stock, and pipeline queues are seeded based on the demand statistics and the lane lead times.【F:backend/app/services/mixed_game_service.py†L1997-L2137】
- This means Complex_SC does inspect the DAG structure at game start, satisfying the requirement to determine how demand should traverse the network before rounds begin.

## Round processing and demand propagation
- `start_new_round` recalculates lane views and builds a `processing_order` by reversing the stored `node_sequence`, ensuring that market demand nodes are handled first and upstream suppliers last. Matured orders are popped from each node’s information queues so that demand is pushed upstream in the correct topological order before shipments are scheduled downstream.【F:backend/app/services/mixed_game_service.py†L2241-L2432】
- Shipment scheduling then walks the forward sequence to push allocations downstream while maintaining arrival queues and shipment pipelines per lane, giving the engine full visibility into in-transit material by lane.【F:backend/app/services/mixed_game_service.py†L2600-L2699】

## Agent invocation
- `process_ai_players` retrieves the same `node_sequence` and iterates over nodes from downstream to upstream (`reversed(node_sequence)`) so that when an agent is called it can observe the orders placed by all of its immediate downstream partners in the current round.【F:backend/app/services/mixed_game_service.py†L2990-L3047】
- For each node the service groups players by canonicalised node key and attempts to instantiate an `AgentType` from that key before dispatching to the configured policy/LLM strategy. Because `AgentType` only defines the classic four beer-game roles (retailer, wholesaler, distributor, manufacturer), any Complex_SC role name outside that set (for example an individual supplier or plant) will raise a `ValueError` before an agent call is made.【F:backend/app/services/mixed_game_service.py†L3053-L3270】【F:backend/app/services/agents.py†L33-L50】
- When debug logging is enabled the service records, for each node visit, the payload supplied to the agent and the resulting decision in a per-round log via `append_debug_round_log`, but this log is written to disk rather than the API payload.【F:backend/app/services/mixed_game_service.py†L3223-L3305】【F:backend/app/simulation/debug_logging.py†L38-L97】

## Reporting surface
- `MixedGameService.get_report` aggregates round history by player role, not by individual node, so the API response exposes per-role orders, inventory, and backlog without revealing the exact traversal order that was used during simulation.【F:backend/app/services/mixed_game_service.py†L3384-L3476】
- The Game Report page rebuilds Sankey data from the shipment history and automatically hides market supply nodes, but it does not expose an interactive filter to focus on specific node types beyond the implicit exclusions baked into `shouldExcludeType`. Node-type summaries in the time-series visualisations are also driven by the canonical role list, so Complex_SC nodes outside the standard four roles are not individually addressable in the UI.【F:frontend/src/pages/GameReport.jsx†L700-L900】

## Findings
1. **Lane ordering is computed correctly** – Complex_SC leverages the topological sort in `_build_lane_views`, and that ordering flows through to both the engine and the agent-processing loop, ensuring demand moves upstream and material downstream according to the DAG.【F:backend/app/services/mixed_game_service.py†L366-L490】【F:backend/app/services/mixed_game_service.py†L2241-L2348】
2. **Agent dispatch is constrained to classic roles** – The call to `AgentType(role_token)` means only retailer/wholesaler/distributor/manufacturer nodes can host agents. Complex_SC introduces additional nodes (plants, region-specific DCs, suppliers), so without remapping them to the four canonical roles the agent invocation will fail before orders are generated.【F:backend/app/services/mixed_game_service.py†L2990-L3270】【F:backend/app/services/agents.py†L33-L37】【F:backend/scripts/create_regional_sc_config.py†L107-L240】
3. **Round-by-round traversal data stays in logs** – Although the backend records per-node debug entries when agents act, those details are written to filesystem logs and are not surfaced through `get_report`, so the in-game report cannot show the node traversal sequence or the individual decisions made during play.【F:backend/app/services/mixed_game_service.py†L3223-L3305】【F:backend/app/services/mixed_game_service.py†L3384-L3476】
4. **UI lacks explicit node-type filters** – The Game Report builds Sankey diagrams and summary tables from aggregated data and excludes market supply nodes by default, but there is no control that lets a user filter the results to a particular node type; everything is grouped under the standard role labels inferred from the data.【F:frontend/src/pages/GameReport.jsx†L700-L900】

## Gaps relative to the requested behaviour
- Agents attached to Complex_SC nodes beyond the four canonical roles cannot be invoked because of the `AgentType` limitation; the configuration needs either role remapping or an expanded enumeration to cover suppliers, plants, and distribution centres explicitly.【F:backend/app/services/mixed_game_service.py†L3053-L3270】【F:backend/app/services/agents.py†L33-L37】
- The runtime does compute and store the traversal order, but the player-facing report endpoint collapses everything to per-role aggregates, so users cannot inspect the actual sequence of node visits or view each ordering decision while a game is in flight.【F:backend/app/services/mixed_game_service.py†L3223-L3305】【F:backend/app/services/mixed_game_service.py†L3384-L3476】
- The front-end Sankey visualisation provides an overview but only applies hard-coded exclusions for market supply/demand; additional UI controls are needed to satisfy the requirement for node-type filters when analysing Complex_SC results.【F:frontend/src/pages/GameReport.jsx†L700-L900】
