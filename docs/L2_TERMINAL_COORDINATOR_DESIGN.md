# L2 Terminal Coordinator — Design Doc (v1 draft)

**Status:** Design, 2026-04-23. Pre-requisite to implementation.
**Layer:** L2 Operational (between L1 TRMs and L3 Tactical planners).
**Author context:** Follows `TMS_DECISION_HIERARCHY.md` §5.
**SCP analog:** SCP's `site_tgnn` (GATv2 + GRU per physical site). Same
architecture family, transport-domain semantics.

---

## 1. What the Terminal Coordinator is

A single agent instance runs **per transportation terminal** (DC,
cross-dock, rail ramp, ocean-port terminal) and modulates the 11 L1
TRMs running at that terminal. Always-on (not scheduled). Inputs
stream in via `HiveSignalBus`; outputs are urgency modulations + local
coordination directives that L1 TRMs consume within their decision
cycles.

It is **not** a planner. It does not produce `transportation_plan`
rows. L3 owns plan production. L2 owns **hub-local urgency + batching
+ sequencing** inside the envelope L3 has committed.

The six canonical L2 decisions are already listed in
`TMS_DECISION_HIERARCHY.md §5.1`; this doc details the contract.

---

## 2. Scope

### In scope
1. Cross-TRM urgency modulation at the hub
2. Carrier-waterfall depth tuning per lane
3. Active shipment batching within shift
4. Dock re-sequencing on appointment slip
5. Yard equipment-pool management
6. Terminal health signal aggregation (upward to L3)

### Out of scope
- Network-level decisions (lanes, carriers) — L3
- Strategic envelope (θ, BSC weights, service-level tiers) — L4
- Per-shipment execution decisions (who gets the tender, what dock slot) — L1
- Cross-hub coordination — L3 sees multiple hubs; L2 is hub-local
- SCP-domain concerns (SKU inventory, safety stock, MO release) — SCP

---

## 3. Agent architecture (target)

### Shape
GATv2 (Graph Attention Network v2) + GRU (Gated Recurrent Unit), per
SCP's proven pattern in `site_tgnn`. Graph is hub-local; nodes are
the terminal's resources; edges encode co-dependence.

### Node types
| Node type | Features |
|---|---|
| **dock_door** | open/closed, appointment queue depth, current appointment status, service time distribution |
| **lane** (outbound from this hub) | queued shipments, mode, recent tender-reject rate, spot availability |
| **lane** (inbound to this hub) | expected arrivals next 4h, ETA confidence P10/P50/P90 |
| **equipment_pool** | trailers/containers by type, status (AVAILABLE/IN_USE/MAINTENANCE), age since last move |
| **carrier_presence** | on-property carriers, time-at-terminal, next-load assignment |
| **shipment_queue** | unassigned shipments by service-level tier, SLA-clock-remaining |
| **TRM_agent** | each of the 11 L1 TRMs as a node, with its current urgency + recent decision mix |

### Edge types
- `dock → shipment_queue` (this dock can serve these shipments)
- `lane → shipment_queue` (this lane accepts these shipments)
- `carrier_presence → lane` (which carriers are ready for which lanes)
- `equipment_pool → lane` (equipment feeding this lane)
- `TRM → resource` (which TRM decides for which node)

### Parameter count target
~25K params (same order-of-magnitude as SCP site_tgnn). Small enough
to be fast (<100ms per inference), large enough to capture local
topology.

### Training regime
1. **BC warmup** from synthetic hub-hour snapshots generated by the
   TMS digital twin (`TMS_TIER3_FIRST_PLAN.md` Phase A).
2. **PPO fine-tune** with a reward that trades off:
   - SLA-miss penalty (shipments that breached service window)
   - Tender-reject rate at this hub
   - Dock utilization (too-low or too-high both penalize)
   - Equipment-imbalance penalty at end of shift
3. **Per-tenant per-hub** checkpoints once real operational data flows.

### Inference cadence
- Hourly scheduled run + event-triggered on HiveSignal arrival (dock
  slip, tender reject, exception escalation, equipment surplus/deficit crossing thresholds).
- Inference latency target: <100ms p95.

## 4. Data contract — inputs

### 4.1 From `HiveSignalBus` (real-time)

Each L1 TRM emits `HiveSignal` on every decision. L2 consumes:

| Signal source | Emit on | L2 uses for |
|---|---|---|
| CapacityPromiseTRM | ACCEPT / REJECT | lane-capacity saturation tracking |
| ShipmentTrackingTRM | ESCALATE | inbound-arrival risk |
| DemandSensingTRM | MODIFY | volume surge indicator |
| ExceptionManagementTRM | any | exception-backlog counter |
| CapacityBufferTRM | MODIFY | lane stress signal |
| FreightProcurementTRM | REJECT (all-waterfall exhausted) | carrier-exhaustion flag |
| BrokerRoutingTRM | ACCEPT broker | fallback-frequency tracker |
| DockSchedulingTRM | RESCHEDULE | dock churn indicator |
| LoadBuildTRM | low-utilization | consolidation opportunity |
| IntermodalTransferTRM | ACCEPT mode-shift | mode-mix dashboard |
| EquipmentRepositionTRM | REPOSITION / HOLD | yard balance delta |

### 4.2 From canonical-state tables (polled hourly / on-demand)

| Source | Purpose |
|---|---|
| `dock_door` + `appointment` | dock queue depth, appointment schedule |
| `load` (filtered to this hub's origin/destination) | shipment-to-load assignment state |
| `freight_tender` (trailing 6h) | rolling tender-reject rate |
| `equipment` (filtered to this hub) | yard equipment inventory |
| `tracking_event` (trailing 4h, this hub's shipments) | inbound ETA updates |
| `lane_profile` (this hub's lanes) | recent tender-reject history, OTRI |
| `tms_shipment` (this hub, status in DRAFT/PLANNED/TENDERED) | unassigned-shipment queue |
| `carrier_scorecard` | carrier reliability context |

### 4.3 From L3 (daily — policy envelope)

| Source | Purpose |
|---|---|
| `transportation_plan` where `plan_version='constrained_live'` and this hub | committed plan for the period; L2 modulates inside this envelope |
| `capacity_target` where this hub's lanes | carrier capacity commitments |
| `tender_commitment` (weekly, per lane) | which carriers L3 committed to this week |

### 4.4 From L4 (monthly — policy parameters θ)

| Source | Purpose |
|---|---|
| `policy_parameters` row for this tenant (L4 doc §5 to be built) | BSC weights, service-level tier definitions, mode-mix targets |

## 5. Data contract — outputs

### 5.1 Urgency modulation (per TRM)

New table: `terminal_urgency_override`

| Column | Type | Comment |
|---|---|---|
| `id` | bigint | |
| `tenant_id` | int | FK tenants.id, NOT NULL |
| `config_id` | int | FK supply_chain_configs.id, NOT NULL |
| `hub_site_id` | int | FK site.id |
| `trm_type` | varchar(30) | one of 11 canonical TMS TRM types |
| `urgency_multiplier` | float | range [0.5, 2.0]; applied to TRM's baseline urgency |
| `expires_at` | timestamp | soft TTL, default +1h |
| `reason` | text | short reasoning from the coordinator |
| `created_at` | timestamp | default now() |

L1 TRMs read the latest un-expired row for their `(hub, trm_type)` and
multiply their baseline urgency. Absent row ⇒ multiplier = 1.0.

### 5.2 Carrier-waterfall depth override (per lane)

New table: `lane_waterfall_override`

| Column | Type | Comment |
|---|---|---|
| `id` | bigint | |
| `tenant_id` | int | |
| `config_id` | int | |
| `hub_site_id` | int | this hub |
| `lane_id` | int | FK transportation_lane.id |
| `mode` | varchar(20) | |
| `waterfall_depth` | smallint | how many carriers to tender before falling to spot |
| `expires_at` | timestamp | |
| `reason` | text | |

### 5.3 Shipment batching directives

Consumed by L1 LoadBuildTRM via the HiveSignalBus. No new table —
payload rides in `HiveSignal.context_data`:

```json
{
  "directive": "BATCH_HINT",
  "hub_site_id": 42,
  "shipment_ids": [101, 102, 103],
  "suggested_load_group": "G-2026-04-23-17-01",
  "reason": "Dock slot 14:00-15:30 has capacity; consolidate these 3",
  "expires_at": "2026-04-23T13:30:00Z"
}
```

### 5.4 Dock re-sequencing (event-driven)

Consumed by L1 DockSchedulingTRM via `HiveSignal`. Payload:

```json
{
  "directive": "DOCK_RESEQUENCE",
  "dock_door_id": 7,
  "new_schedule": [
    {"appointment_id": 301, "slot_start": "14:30"},
    {"appointment_id": 305, "slot_start": "14:45"}
  ],
  "reason": "Appointment 299 ETA-slip 35min cascades downstream",
  "cascade_to_appointments": [301, 305, 309]
}
```

### 5.5 Yard equipment placement

Consumed by L1 EquipmentRepositionTRM. Payload:

```json
{
  "directive": "YARD_PLACEMENT",
  "hub_site_id": 42,
  "equipment_moves": [
    {"from_zone": "DROP_YARD_A", "to_zone": "LOADING_LANE_3", "equipment_type": "DRY_VAN", "qty": 2}
  ],
  "reason": "Loading lane 3 demand in next 2h exceeds current position",
  "time_window": "next_2h"
}
```

### 5.6 Terminal-health signal (upward to L3)

New table: `terminal_health_signal`

| Column | Type | Comment |
|---|---|---|
| `id` | bigint | |
| `tenant_id` | int | |
| `config_id` | int | |
| `hub_site_id` | int | |
| `timestamp` | timestamp | |
| `composite_health` | float | 0.0-1.0 (0 critical, 1 nominal) |
| `dock_utilization_pct` | float | |
| `tender_reject_rate_1h` | float | |
| `exception_backlog_count` | int | |
| `equipment_imbalance` | float | signed; + = surplus, − = deficit |
| `sla_miss_rate_1h` | float | |
| `trend_7d` | varchar(20) | IMPROVING / STABLE / DEGRADING |
| `active_overrides_count` | smallint | |

L3 planners poll this table; when `composite_health < 0.5` persists
for 2 hours, L3 Integrated Balancer re-solves the Constrained Balanced
Plan for this hub's lanes.

## 6. Implementation phasing

### Phase 0 — prerequisites (blockers)
- `TMS_TIER3_FIRST_PLAN.md` Phase A (TMS digital twin) must be live
  enough to generate hub-hour snapshots for BC training.
- L4 `policy_parameters` schema (separate design doc) must exist
  so L2 can read θ inputs.
- Item 1.13 Stage B (SCP-fork services extraction) must be complete
  so we don't carry SCP-domain HiveSignal semantics into L2 training.

### Phase 1 — scaffolding (2 weeks)
1. Create the four output tables + one migration.
2. Scaffold `services/powell/terminal_coordinator_service.py` — pure
   Python class, no ML yet.
3. Extend L1 TRMs to consume `terminal_urgency_override` +
   `lane_waterfall_override` in their state-builders. Default to
   multiplier = 1.0 / no override when absent.
4. Write a **deterministic heuristic** Terminal Coordinator: hard
   thresholds drive each of the 6 outputs. Ship this first.

### Phase 2 — data ingestion (2 weeks, parallel with Phase 1)
1. Emit `HiveSignal` from every L1 TRM (currently L1 TRMs log but
   don't emit hive signals — wire to bus).
2. Build the hub-hour snapshot extractor (same shape as SCP's,
   transport features).
3. Aggregate historical shipment / load / tender / tracking_event
   data into training corpora.

### Phase 3 — GATv2+GRU training (3-4 weeks)
1. Define graph construction: hub nodes + features, edge types.
2. Define reward function: SLA-miss + reject-rate + dock utilization
   + equipment-imbalance.
3. BC warmup on hub-hour snapshots with heuristic teacher.
4. PPO fine-tune on twin-rolled-out episodes.

### Phase 4 — rollout (2 weeks)
1. Feature flag `L2_TERMINAL_COORDINATOR_MODE` with values
   `heuristic` (Phase 1 scaffolding), `trm` (Phase 3 trained agent),
   `off` (no overrides — L1 TRMs run bare).
2. Per-tenant per-hub enablement.
3. Monitor `terminal_health_signal` + override counts + outcome KPIs.

Total Phase 1-4: ~9-11 weeks of focused work once the prerequisites
are in place.

## 7. Open questions

1. **Hub identity.** TMS `Site` has `type='CROSS_DOCK'` / `'DC'` /
   `'TERMINAL'` today. Does every Site get a Terminal Coordinator, or
   only a curated subset (e.g., ≥N loads/day)? Probably the latter —
   adds a `terminal_coordinator_enabled: bool` column on Site.

2. **Cross-hub signals.** A Coordinator at hub A notices a surge
   that will cascade to hub B in 4h. Does A ping B directly, or does
   it escalate to L3? Proposal: **L3 only**. L2 is hub-local.
   Cross-hub coordination goes through L3's Constrained Balanced
   Plan re-solve.

3. **Override lifecycle.** 1-hour TTL default, but do we let the
   coordinator extend? What's the right lifecycle model — fire and
   forget, or maintain a persistent state the coordinator re-evaluates?
   Proposal: fire + forget with short TTL (15-60 min depending on
   decision class). Coordinator re-fires if condition persists.

4. **Training data availability.** Phase 3 is blocked on ≥4 weeks of
   hub-hour snapshots. For greenfield tenants, fall back to Phase 1
   heuristic until we have enough real data.

5. **Conflict with L3.** What if L3's `constrained_live` plan says
   carrier X gets the tender but L2 has a spot-rate surge making
   broker a better choice? Proposal: L2 can override **if** the
   override stays inside L3's cost guardrails (e.g., ≤10% cost
   delta). Beyond that, L2 escalates to L3 for re-plan.

6. **One coordinator or one per tenant?** Given ~N hubs per tenant
   and ~M tenants, running O(N×M) coordinator instances could get
   expensive. Proposal: one instance per `(tenant, hub)` pair, with
   hot-cold tiering — active hubs (>10 shipments/hour) run full
   inference; cold hubs only on event triggers.

---

## 8. Non-goals

- **Don't make the coordinator a planner.** No `transportation_plan`
  rows. L3 owns that.
- **Don't make it cross-hub.** Cross-hub flows through L3.
- **Don't model SKU inventory.** That's SCP.
- **Don't duplicate TRM logic.** L2's outputs modify TRM **context**,
  not TRM **decisions**. Each TRM still makes its own decision using
  its own heuristic/model.
- **Don't auto-enable.** Phase 4 rolls out per-hub with explicit
  opt-in.
