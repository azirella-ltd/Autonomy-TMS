# Supply Chain Demand Flow and Reporting Analysis

## Demand propagation from Market Demand to Market Supply
- Each market demand definition couples an item distribution to a specific market node. `_resolve_market_node_key` and `_build_market_item_profiles` scan `config.market_demands`, resolve the correct downstream node for every entry, and translate the configured demand pattern into a steady-state baseline per item, so every Market Demand node maintains its own item mix and magnitude.【F:backend/app/services/mixed_game_service.py†L943-L1027】
- `_build_lane_views` turns the explicit lane list into shipment and order adjacency maps while normalising node keys, inferring missing market endpoints, and recording per-lane order/supply lead times. Because each Market Demand node’s downstream lane terminates at its owning distribution centre, the resulting `orders_map` only contains the DCs authorised to satisfy that node, and the `lane_lookup` carries the correct timing data for every hop.【F:backend/app/services/mixed_game_service.py†L618-L755】
- When a game starts, `MixedGameService.start_game` seeds the engine by pushing the market item profiles upstream through the `orders_map`, so every node inherits the exact finished-goods mix demanded downstream. The same routine applies `_resolve_lane_order_delay`/`_resolve_lane_ship_delay` to bake the lane-level order and shipment delays into each node policy, ensuring every future order placed against a lane uses that lane’s lead-time characteristics instead of a generic average.【F:backend/app/services/mixed_game_service.py†L3182-L3251】

## Order timing, lead times, and what Market Supply should see
- Customer demand matures according to the configured order lead time, including same-round maturity when a lane explicitly sets `order_leadtime` to 0. `_inject_market_demand` reads the supplier’s `order_leadtime` from the supply-chain config, applies `lead = max(0, order_leadtime)`, and stamps each request with `due_round = current_round + lead`; orders remain in the supplier’s `inbound_demand` until their due round arrives. For the Default TBG setup, seeding enforces `order_leadtime = 1` on every arc, so Market Demand orders do not mature until the following round.【F:backend/app/services/mixed_game_service.py†L1611-L1655】【F:backend/scripts/seed_default_group.py†L368-L407】【F:backend/scripts/seed_default_group.py†L933-L966】
- `_preprocess_queues` only moves orders from `inbound_demand` into `matured_orders` when `due_step <= current_round`, so the replenishment logic that computes a node’s demand never sees future-dated orders. This prevents upstream demand from aggregating in the creation round even if a downstream node submitted multiple requests at once.【F:backend/app/services/mixed_game_service.py†L1500-L1534】
- When a node places replenishment orders upstream, those orders are scheduled with that lane’s configured `order_lead_time` and the corresponding shipment delay. Shipments are tracked as `inbound_supply_future` until they mature to `inbound_supply`. Demand tallies for the supplier are incremented in the same round only when `due_round <= current_round`, so Market Supply reflects whatever the config dictates—for the TBG config (order leadtime 1, supply leadtime 2), Market Supply sees only the Manufacturer’s prior-round order rather than the sum of every echelon’s current-period requests.【F:backend/app/services/mixed_game_service.py†L1888-L1930】【F:backend/main.py†L2765-L2833】【F:backend/main.py†L2824-L2931】

## Manufacturer BOM explosion through component suppliers
- `_derive_component_metadata` now normalises the configured bill of materials instead of auto-generating synthetic components, preserving the master-data definition of which components (and quantities) each finished good requires.【F:backend/app/services/mixed_game_service.py†L1077-L1120】
- When a manufacturer has a single upstream supplier, the routine records that supplier as the source for every component in the manufacturer’s BOM and seeds a baseline entry for each component on that supplier so production can consume the correct materials.【F:backend/app/services/mixed_game_service.py†L1122-L1135】

## Upstream order calculation and downstream supply arrival
- During a round, `_start_new_round` collects each node’s new demand and divides the order quantity across the upstream parents listed in `orders_map`. For every upstream lane it snapshots the receiving node’s information queue based on that lane’s order lead time, so a Market Demand node’s orders enter the exact DC pipeline dictated by the lane definition rather than being sprayed across unrelated nodes.【F:backend/main.py†L2765-L2833】
- Step three of the same routine iterates over the topological node sequence to schedule shipments. It recomputes available inventory after applying immediate arrivals, allocates the shipped quantity across each downstream lane based on the orders that lane placed, and then either appends the load to the downstream lane’s arrival queue with the correct `supply_leadtime` or applies it immediately if the lane delay is zero. This is how supply physically travels along the SC DAG from Market Supply to Market Demand while respecting lane-specific transit times.【F:backend/main.py†L2824-L2931】

## Game Report filtering and reporting fixes
- `_ensure_node_type_definitions` mirrors only the node types found in the supply-chain configuration (plus the mandatory market endpoints), normalising legacy aliases so the API no longer exposes phantom roles such as Wholesaler when they are absent from the DAG. All derived label maps and sequence hints are stored back on the config for downstream consumers.【F:backend/main.py†L1554-L1654】
- Sankey shipment reconstruction now requires the configuration-provided node-type ordering and raises an error if the DAG cannot be traversed, replacing the old hard-coded retailer/wholesaler fallback so flows always mirror the actual supply-chain topology.【F:backend/main.py†L116-L218】
- The Game Report header highlights a single hero metric – total Market Demand orders satisfied – and formats it with grouped thousands separators via the shared `quantityFormatter`, satisfying the requirement to use commas for every reported number.【F:frontend/src/pages/GameReport.jsx†L340-L388】【F:frontend/src/pages/GameReport.jsx†L2658-L2691】【F:frontend/src/pages/GameReport.jsx†L3872-L3882】
- The node-type multi-select handler now copies the MUI `value` payload into a mutable `nextValues` array, strips the `"all"` sentinel when additional node types are chosen, and deduplicates the result before updating state. This prevents the “unable to select a node type” failure that occurred when the code attempted to mutate the immutable `value` array returned by the Select component.【F:frontend/src/pages/GameReport.jsx†L3645-L3667】
