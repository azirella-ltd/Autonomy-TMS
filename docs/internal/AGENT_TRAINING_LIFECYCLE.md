# Agent Training Lifecycle — Technical Reference

> **Audience**: Engineers, data scientists, solution architects.
> Precise technical documentation of the full training pipeline: Phase 1 (BC),
> Phase 2 (Offline RL / VFA fine-tuning), and the CDC → Relearning autonomous loop.
>
> **Also see**: [UNIFIED_TRAINING_CORPUS.md](architecture/UNIFIED_TRAINING_CORPUS.md) —
> Authoritative reference for the training data pipeline. As of April 2026, all four
> planning layers (Strategic, Tactical, Operational/Site, Execution/Role) train on a
> single unified corpus anchored on the ERP baseline, with bottom-up aggregation from
> TRM decisions up to strategic policy parameters. This supersedes the previous
> architecture where each layer had its own synthetic data pipeline.

---

## Overview

**Critical distinction**: TRM is a model *architecture*; RL is a *training method* applied
to that architecture. These are not alternatives. The same TRM checkpoint is the output
of both Phase 1 (behavioral cloning) and Phase 2 (RL fine-tuning). The architecture does
not change between phases — only the weights change.

The same distinction applies at all agent tiers:

| Agent Tier | Architecture | Training Method |
|------------|-------------|-----------------|
| Strategic (S&OP) | GraphSAGE | CFA via Differential Evolution (weekly, not RL) |
| Tactical (Network) | Temporal GNN | Supervised + PPO (Phase 2) |
| Operational (Site) | Site tGNN (GATv2+GRU, ~25K params) | Behavioral cloning + PPO |
| Execution (Role) | TRM (2-layer transformer, 7M params) | BC (Phase 1) + Offline RL VFA (Phase 2) |

## Phase 1: Behavioral Cloning (Warm-Start)

### What it is
Supervised learning from a corpus of (state, decision) pairs derived from:
- Historical engine decisions (from `ScenarioUserPeriod` records and provisioning simulation)
- Synthetic curriculum data generated by `SyntheticTRMDataGenerator`
- Expert planner decisions where available

### Why it exists
RL on a randomly initialized network over a large state-action space is sample-inefficient
and unstable. BC warm-start provides a weight initialization that is already in a useful
region of the policy space, dramatically reducing the number of RL steps required in
Phase 2 to reach competent performance.

### Training objective
Standard cross-entropy loss (discrete actions) or MSE (continuous quantity outputs):
```
L_BC = -Σ [y_expert * log π_θ(a | s)]
```
where `π_θ` is the TRM policy and `y_expert` is the one-hot expert action.

### Data requirements
- **Phase 1 has no data minimum** — it always runs using synthetic curriculum data from
  `SyntheticTRMDataGenerator`. No live decision history is required.
- Generated by: `TRMSiteTrainer.train_phase1()` with 3 sub-phases of increasing complexity
  (simple → moderate → full complexity), 20 epochs default
- **Default volume**: 50,000 samples per sub-phase × 3 sub-phases = 150,000 per signal phase

### Data Volume: "Learning by Watching" (Stöckl 2021)

Our data volume targets are informed by Stöckl's "Watching a Language Model Learning Chess"
(RANLP 2021, [ACL Anthology](https://aclanthology.org/2021.ranlp-1.148/)), which studied how
GPT2 models of varying sizes learn structured decision-making from supervised observation
of expert play. The core finding: **for structured decision tasks, data volume matters more
than model size**. A smaller model with sufficient data outperforms a larger model with
insufficient data on rule learning and generalization.

The analogy to TRM training is direct: TRMs learn supply chain decision-making by "watching"
deterministic engines make expert decisions across thousands of synthetic scenarios — the same
paradigm as language models learning chess by observing expert games. The critical question
is not whether the 7M-parameter TRM architecture is powerful enough (it is), but whether we
expose it to enough diverse (state, expert_action) pairs to learn the underlying decision
rules rather than just memorizing surface patterns.

**Three key lessons applied:**

1. **Standard loss is misleading.** Stöckl showed that models achieving low perplexity still
   made illegal chess moves. In our context: low BC training loss does not mean the TRM
   makes correct decisions on novel states. That is why `train_phase1()` now automatically
   runs `evaluate_3tier()` after training — measuring correct-decision-rate on held-out and
   adversarial states, not just training loss.

2. **Three-tier evaluation.** Every Phase 1 training run now reports:
   - **Tier 1 (Memorization)**: Accuracy on training data (should be >95%)
   - **Tier 2 (Generalization)**: Accuracy on held-out states from same distribution (target >85%)
   - **Tier 3 (Rule Learning)**: Accuracy on adversarial/edge-case states — zero inventory,
     demand spikes, capacity extremes (target >70%)

3. **Data volume thresholds.** For 7M-param models (extrapolated from Stöckl + Kaplan 2020
   scaling laws):

   | Regime | Samples | Behavior |
   |--------|---------|----------|
   | Insufficient | <10K | Memorization only, no generalization |
   | Medium | 50K–150K | Rule learning begins, handles typical states |
   | Robust | 500K+ | Generalizes to edge cases and novel combinations |

   Phase 1 generates 50K × 3 sub-phases × 3 signal phases = **450K samples** per TRM
   (the "robust" regime). The `StochasticCurriculumWrapper` supports a `multiplier`
   parameter that generates M independent Monte Carlo draws — each with a different random
   seed for both the base curriculum and the stochastic distribution overlay — enabling
   further scaling to millions of samples when needed.

### What Phase 1 produces
A checkpoint that reproduces synthetic curriculum decisions. The agent is competent but
capped at curriculum-level performance — it cannot discover policies that outperform the
training data. Phase 2 is required to go beyond this baseline.

### Provisioning step
`trm_training` in `ConfigProvisioningStatus.STEPS`. Runs BC for all 11 TRM types for
the target config. Always succeeds on first provision since it requires no live data.

## Phase 2: Offline RL / VFA Fine-Tuning ("On the Job")

### What it is
Value Function Approximation (VFA) — Powell's policy class for sequential decision
problems where a model of the environment is partially known. Implemented as Offline RL
using Temporal Difference (TD) learning from the decision-outcome pairs accumulated in
the `powell_*_decisions` tables.

### Why it differs from Phase 1
Phase 1 minimizes imitation error (distance from expert action). Phase 2 maximizes
*expected cumulative reward* (Powell objective function F(x, W)). The reward signal is:

```
r(s, a, s') = w_cost * Δcost + w_service * Δservice_level + w_inv * Δinventory_turns
```

where weights `w_*` are configured via `EconomicCostConfig` in `trm_trainer.py` and are
denominated in dollars (Lokad economic loss function approach — see POWELL_APPROACH.md §5.18).

### The key distinction from live RL
Phase 2 is **Offline RL** (also called batch RL): the agent is not interacting with a live
environment. Instead, it learns from a fixed dataset of (state, action, reward, next_state)
tuples already collected in `powell_*_decisions`. This avoids the instability and safety
risks of online RL in production while still enabling the agent to learn counterfactual
improvements.

Offline RL limitation: the agent cannot explore states not represented in the training
corpus. This is why the BC warm-start (Phase 1) is essential — it ensures the initial
policy visits a wide state distribution, giving Offline RL a richer corpus to learn from.

### TD update rule (simplified)
```
Q(s, a) ← Q(s, a) + α * [r + γ * max_a' Q(s', a') - Q(s, a)]
```
The TRM's value head approximates `Q(s, a)`. The 3-step recursive refinement in the TRM
architecture provides additional gradient signal at intermediate steps (deep supervision),
stabilizing the TD update across the recursion depth.

### Data requirements
- **Phase 2** (`train_phase2`): minimum 500 expert decision records with outcomes
- **Phase 3** (`train_phase3`): minimum 1,000 replay buffer entries with outcomes
- Outcome columns populated by: `OutcomeCollectorService.collect_trm_outcomes()` (hourly at :32)
- Feedback horizons: ATP=4h, inventory_buffer=24h, po_creation=7d (see `outcome_collector.py`)

### Implementation
- `TRMSiteTrainer.train_phase2()` — fine-tunes on human expert decisions from powell_*_decisions
- `TRMSiteTrainer.train_phase3()` — TD/CQL Offline RL on replay buffer with Bellman updates
- Phase 3 uses Conservative Q-Learning (CQL) penalty to prevent Q-value overestimation on
  out-of-distribution state-action pairs (critical for offline stability)
- `CDCRetrainingService.execute_retraining()` triggers Phase 2/3 re-runs when CDC fires

## Phase 3: CDC → Relearning Loop (Continuous Improvement)

### What it is
An autonomous, scheduled pipeline that monitors agent performance, detects distribution
shift (concept drift), and triggers retraining without human intervention. This is the
"continuous improvement" loop.

### Architecture

```
powell_*_decisions tables (11 types)
        │
        ▼ hourly at :32
OutcomeCollectorService.collect_trm_outcomes()
        │ fills actual_outcome, actual_reward columns
        ▼ hourly at :35
CDTCalibrationService.calibrate_incremental()
        │ updates conformal risk bounds: P(loss > τ) per TRM type
        ▼
All TRM decisions carry risk_bound (CDT guarantee)
        │
        ▼ continuous
CDCMonitor.check_and_trigger()
        │ fires when metric deviation exceeds threshold in powell_cdc_thresholds
        ├─ REPLAN_LOCAL  → signal SiteAgent
        ├─ VERTICAL_OPERATIONAL → EscalationArbiter (tGNN re-run)
        └─ VERTICAL_STRATEGIC  → EscalationArbiter (S&OP re-optimize)
        │
        ▼ every 6h at :45
CDCRetrainingService.evaluate_retraining_need()
        │ conditions: ≥100 new experiences + CDC trigger + cooldown elapsed
        ▼ if conditions met
CDCRetrainingService.execute_retraining()
        │ runs Offline RL (Phase 2) on accumulated decision-outcome pairs
        │ regression guard: new model must beat current on held-out validation
        ▼ if regression guard passes
SiteAgent.reload_model(checkpoint_path)
```

### CDC thresholds (from `powell_cdc_thresholds`)
Seven monitored conditions in `CDCMonitor.check_and_trigger()`:
1. `demand_deviation`: demand ±15% from forecast
2. `inventory_ratio_low`: on-hand < 70% of target
3. `inventory_ratio_high`: on-hand > 150% of target
4. `service_level_drop`: service level 5% below target
5. `lead_time_increase`: lead time +30% above baseline
6. `backlog_growth_days`: backlog growing for 2+ consecutive days
7. `signal_divergence`: CDC signal bus divergence above threshold
(Also monitors `supplier_reliability_drop` as an 8th condition.)

### Regression guard
New checkpoints are compared against the **current checkpoint's training loss**, not a
formal held-out validation set. Deployment requires:
- New checkpoint loss improvement does not regress > 10% vs current checkpoint
- This is a lightweight safeguard; it prevents clearly worse models from deploying but
  does not provide a formal out-of-sample coverage guarantee. A formal holdout evaluation
  is a known gap — see improvement backlog.

### Escalation path for structural drift
If CDC triggers persist across 3+ consecutive retraining cycles without improvement,
`EscalationArbiter` escalates to a higher tier. The execution-level agent cannot fix a
structural supply chain change (e.g., a supplier permanently doubling lead times) — that
requires the tactical or strategic agent to re-optimize policy parameters. See
`docs/internal/ESCALATION_ARCHITECTURE.md` for the vertical escalation protocol.

## Override Learning Integration

Human planner overrides feed into the training loop via three mechanisms:

### 1. Bayesian posterior update (immediate)
Every override increments `Beta(α, β)` in `override_effectiveness_posteriors` for the
`(user_id, trm_type)` pair. E[p] = α/(α+β) tracks whether this user's overrides for this
decision type historically improve outcomes.

### 2. Training weight modulation (Phase 2)
Override decisions with high `E[p]` receive higher weight in the Offline RL loss:
```
training_weight = 0.3 + 1.7 * E[p]   # range: [0.3, 2.0]
```
Users whose overrides consistently improve outcomes have outsized influence on the next
TRM checkpoint.

### 3. Counterfactual reward injection (Tier 1/2 decisions)
For ATP, forecast adjustment, and quality decisions (Tier 1 — analytical counterfactual),
the reward the agent's *original recommendation* would have earned is computed and
injected into the training corpus as a (state, agent_action, counterfactual_reward) tuple.
This teaches the agent about outcomes in the override state, not just the accepted-decision
state. See `outcome_collector.py` and `OVERRIDE_EFFECTIVENESS_METHODOLOGY.md`.

## Site tGNN Training (Operational Level)

The Site tGNN is trained separately from the execution role agents, using a 3-phase
pipeline that mirrors the TRM pipeline but at the site level:

| Phase | Method | Target |
|-------|--------|--------|
| Phase 1 | BC from MultiHeadTrace records | Reproduce cross-TRM urgency coordination |
| Phase 2 | PPO with site-level BSC reward | Improve cross-TRM trade-off resolution |
| Phase 3 | Shadow calibration from live data | Calibrate urgency magnitude to actual outcomes |

Training check: every 12h at :50. Requires `enable_site_tgnn=True` in `SiteAgentConfig`.
Implementation: `SiteTGNNTrainer` in `site_tgnn_trainer.py`.

---

## Tactical Agent Training (Network Level)

The Demand, Supply, and Inventory tGNNs are trained via the RLDashboard:

- **Demand tGNN**: Supervised on forecast error + PPO for consensus demand accuracy
- **Supply tGNN**: Supervised on supply plan optimality (cost + service) + PPO
- **Inventory tGNN**: Supervised on buffer effectiveness + PPO for holding cost vs. stockout

PPO is used (not Offline RL) because these agents interact with the digital twin simulation
environment, which provides a rich online signal. The simulation is the "environment" in
the RL sense — it is not real production data.

## Strategic Agent (S&OP Level)

The S&OP GraphSAGE agent does **not** use RL. It uses Cost Function Approximation (CFA)
via Differential Evolution, a gradient-free optimization method:

```
θ* = argmin_θ E_W [C(X^π(S|θ), W)]
```

Policy parameters θ (safety stock multipliers, service level targets, sourcing priorities)
are optimized over Monte Carlo scenarios W sampled from the stochastic simulation.
This runs weekly on Sunday at 04:00 via `_run_cfa_optimization()` in `relearning_jobs.py`.

CFA is appropriate at the strategic level because:
1. The decision space is continuous and low-dimensional (θ parameters, not discrete actions)
2. The feedback horizon is weekly/monthly — too long for TD learning to be practical
3. The objective is smooth enough for Differential Evolution to converge reliably

## Relationship Between Tiers During Training

The tiers are not independent — they form a training dependency chain:

```
S&OP GraphSAGE CFA
    │ produces policy parameters θ (safety stock multipliers, SS targets)
    ▼
Tactical tGNNs use θ as part of their state input
    │ produce priority allocations and demand directives
    ▼
Site tGNNs use tGNN outputs as urgency modulation signals
    │ modulate TRM urgency vectors
    ▼
Execution TRMs use site tGNN urgency as part of their state input
```

During provisioning, this is why the step dependencies exist:
- `demand_tgnn` depends on `sop_graphsage` (needs policy parameters)
- `inventory_tgnn` depends on `supply_tgnn` (needs supply baseline)
- `trm_training` depends on all tactical agents (needs their outputs as state features)
- `site_tgnn` depends on `decision_seed` (needs live TRM decisions as training signal)

## Stochastic Variables in Training Data Generation

All synthetic training data used across every agent tier is generated from a single shared
module: **`backend/app/services/powell/training_distributions.py`** (`D` class).

### Why a Single Module

Prior to this module, each oracle used inline `rng.uniform()` or `rng.normal()` calls with
independently chosen distribution types and parameter ranges. This caused **cross-tier
distribution drift**: the same conceptual variable (e.g., lead time) was sampled differently
at different tiers, so the S&OP GraphSAGE was optimising policy parameters against
distributions the TRMs would never actually encounter. `training_distributions.py` eliminates
that inconsistency — every oracle imports from `D`.

### Distribution Choice Criteria

| Distribution | When to use | Examples |
|---|---|---|
| **Triangular** | Operational state variable that clusters around a known typical value. Mode encodes domain knowledge. PERT standard for bounded expert estimates. | Inventory, capacity, lead time (scenario level), demand forecast |
| **Uniform** | Equal coverage of full range desired, OR no domain-informed prior about which value is "most common" (varies enormously across industries). | Unit cost, ordering cost, service level target, criticality score |
| **LogLogistic** | Within-simulation realisation of physically right-skewed processes. Fat right tail captures supplier disruptions better than LogNormal. | Realised lead time (MC loop) |
| **Normal** | Realised demand within a simulation. Demand is approximately symmetric around the forecast; CLT applies. | Realised demand (MC loop) |
| **Bernoulli** | Binary event flags. Probability scales with curriculum phase (more disruptions at Phase 3). | Quality hold, maintenance due, ATP shortfall |
| **Poisson** | Count variables. Correct model for rare-event counts; better than Discrete Uniform. | Open order exceptions |

### Phase Variance Schedule

Variance intensity scales with curriculum phase:

| Phase | `variance_pct` | Regime |
|-------|----------------|--------|
| 1 | 0.15 | Tight — clear signals, stable state |
| 2 | 0.40 | Moderate — trade-offs, seasonal swings |
| 3 | 0.75 | High — disruptions, cascading failures |

For Triangular jitter variables, `variance_pct` controls the distribution width around the nominal
mode. Right-skewed variables (demand, inventory) use `right_factor > left_factor` so that
upside spikes are larger than downside troughs.

#### Operational Level — Site tGNN (site_tgnn_oracle.py)

The Site tGNN sees the same site state as the TRMs above. The oracle generates urgency
adjustment labels by running all 11 deterministic engines, detecting resource conflicts,
and resolving via 5-tier priority rules:

1. Customer commitments (ATP, Order Tracking)
2. Production safety (MO Execution, Quality)
3. Supply continuity (PO Creation, TO Execution)
4. Lateral flow (Inventory Rebalancing, Subcontracting)
5. Planning refinement (Forecast Adjustment, Inventory Buffer, Maintenance)

**No additional stochastic variables** beyond the TRM site state variables. The oracle output
is `target_adjustments [11, 3]` = (urgency_delta, confidence_delta, priority_delta) per TRM
node.
